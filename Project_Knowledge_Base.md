# 📚 SuperWorkspace 项目知识库与封存档案

## 🗄️ 档案 1：VMulti 内核级驱动接入计划
目前的方案使用 `User32.dll` API，属于用户层模拟。存在游戏防作弊拦截和 UAC 弹窗拦截的限制。
VMulti 是内核级驱动，能在设备管理器捏造真实的物理触摸屏，实现降维打击。

**唤醒条件**：当强烈要求在电脑玩手游模拟器，或软件向“专业外设”转型时开启。

## 🗄️ 档案 2：虚拟摄像头“绿屏与断流”系统级诊断

**病症表现**：
启用超级摄像头时，电脑微信视频中显示带有红字的“绿黄相间条纹测试图”。Scrcpy 引擎抛出 `Device disconnected` 并在数秒后闪退。

**物理层原因分析**：
1. **绿屏的本质是“失联”**：这并不是 WPF 代码在渲染绿色。这是 `UnityCapture` 虚拟驱动底层的 `No Signal Fallback`（无信号占位图）。当 C# 抓取循环因为目标窗口消失而**停止向共享内存（UnityCapture1）注入数据并停止更新帧序号时**，内核驱动就会判定流已断开，进而显示绿屏警告微信。
2. **窗口消失的元凶是“内核杀后台”**：由于安卓 14 系统（尤其是高度定制的 MagicOS / HyperOS）在底层进行了极为严格的相机权限隔离。当手机处于**息屏、人脸识别轮询唤醒、后台挂有占用麦克风/相机的应用**时，系统发现一个权限为 ADB (UID 2000) 的底层进程试图独占流，为了防止黑客后台偷拍，系统的资源管理器会产生锁冲突，直接从系统内核层强行踢掉 ADB 的 USB 连接。这直接导致了 `Device disconnected`。

**标准规避方案 (操作 S.O.P)**：
1. 取消勾选“自动息屏”，保持手机屏幕长亮并停留在主桌面。
2. 确保人脸识别等底层传感器未被频繁唤醒。
3. 确保彻底关闭微信等占用虚拟驱动的软件，先开启我们的采集端，等窗口出现且画面稳定后，再开启微信的视频流捕获。

## 🗄️ 档案 3：智能控制台中枢 (Feature 6) 杀手级增量开发规划

本规划旨在将单纯的“被动按键面板”升级为“主动感知的数据闭环外设”。

**三大核心底层技术维度**：
1. **主动感知 (Context-Awareness)**：C# 端调用 Win32 API (`SetWinEventHook` 监听 `EVENT_SYSTEM_FOREGROUND`) 实时捕捉电脑前台进程（如 Word、原神）。通过 WebSocket 主动下发指令，平板端无感自动横滑切换到对应的快捷键场景。
2. **数据闭环 (Data Feedback)**：C# 后台新增 `SystemMonitor` 线程，高频抓取 CPU、Docker 容器状态或剪贴板翻译内容，转为 JSON 推送给平板，前端根据 ID 动态替换 DOM 文本、颜色或渲染图表（实现带状态的活按钮）。
3. **硬件级交互 (Hardware Fusion)**：深度挖掘 HTML5 硬件 API。按键按下触发 `navigator.vibrate` 实现物理震动反馈；采集 `DeviceOrientationEvent`（陀螺仪）数据通过 60Hz 频次回传 C# 映射为 vJoy 虚拟摇杆，实现体感赛车/飞行模拟。

**未来拓展场景清单 (The Next Big Things)**：
- **场景 8：智能环境与 PC 健康监控 (Command Center)**
  - *动态资源雷达*：炫酷仪表盘实时显示 CPU/GPU 频率、温度，一键切换性能模式。
  - *多路音轨调音台*：独立滑动调节不同 App 的音量（分离浏览器与游戏声音）。
  - *智能家居联动*：切入游戏模式时，自动调用米家/HomeAssistant API 调暗房间物理灯光。
- **场景 9：AI 全球脑与知识库 (AI Copilot Hub)**
  - *剪贴板实时 AI 翻译*：电脑端复制外文文献，平板端瞬间自动显示中文翻译和摘要提取。
  - *语音咒语透传*：对平板说话，AI 生成代码后通过 `PasteText` 闪电剪贴板引擎瞬间注入 PC 编辑器。
- **场景 10：极客外设增强 (Experimental Labs)**
  - *虚拟触控条 (Virtual TouchBar)*：为 PC 追加可滑动的自定义交互条。
  - *体感游戏控制器*：平板化身真·方向盘。
  - *生物识别安全锁*：基于平板与 PC 的蓝牙 RSSI 信号强度，人走自动执行“一键锁屏”。

## 🗄️ 档案 4：控制台 Linux 面板黑屏事件 (已诊断留坑)

**病症表现**：点击“Linux 工具”后，平板屏幕拉下全屏黑透遮罩，但没有显示任何按钮，一片漆黑。

**技术复盘**：DOM 节点层级越界。在组装超长 HTML 字符串时，`<div id='devtools-ui'>` 错误地放置在了 `<div id='overlay-container'>` (全屏遮罩层) 的**外部**。当 JS 触发显示时，黑色的遮罩层占据了 `z-index: 1000` 并铺满全屏，而 Linux 面板虽然被设置为 `display: flex`，但由于未在遮罩层内部，被“黑色玻璃”彻底掩盖。

**处理决定**：暂缓修复，留作前端重构时的经典反面教材。优先推进 Feature 7 大局。

## 🗄️ 档案 5：极客运维面板 (Vim/Tmux) 业务降级与封存

**业务复盘**：在功能 6 (自定义控制台) 中尝试引入了专门针对 Linux 运维的次级面板，包含 Vim 强制保存、Tmux 分屏等复杂的按键连招。

**封存原因**：
1. **前端复杂度溢出**：硬编码的次级面板 DOM 嵌套极易出错（直接引发了档案 4 的黑屏惨案），继续堆砌定制 UI 会彻底破坏当前极简的 JSON 数据驱动架构。
2. **伪需求与替代方案**：这种长尾的极客需求，完全可以让用户自己通过平板上的 `+` 悬浮按钮，配合 `hotkey|Ctrl+B` 或 `run|` 等通用动态指令自行拼装。在源码里写死解析器属于“过度设计 (Over-engineering)”。

**处理决定**：剥离“硬编码”的运维专属 UI 执念，回归“完全数据驱动”。正式封存此专属面板的开发，全军出击，全速推进“功能 7：无缝文件穿越”大局。

## 🗄️ 档案 6：跨端无缝剪贴板 (Feature 7) 方案归档与止损

**业务复盘**：尝试实现 Windows PC 与平板设备之间的“无缝剪贴板双向同步”。
- PC 端：使用 Win32 API `AddClipboardFormatListener` 零延迟监听 `WM_CLIPBOARDUPDATE` 消息，通过 WebSocket 推送。
- 平板端：因 HTTP 协议触发了现代浏览器的安全隐私墙（Secure Context 限制），无法静默写入，故采用顶部浮窗引导用户点击 + `document.execCommand('copy')` 进行优雅降级。

**封存原因**：
1. **产品交互反噬**：将两台高频使用设备的剪贴板强制绑定，在实际场景中极易导致数据的意外覆盖和干扰，反而降低了生产力。
2. **安全隔离机制阻碍**：由于局域网 HTTP 限制，始终需要用户进行多余的点击确认，未能达到“无感穿越”的极致体验标准。

**处理决定**：及时止损，放弃将两端系统剪贴板强行融合的做法。未来重构时，跨端数据传输应调整为：
- 采用“独立中转站 (DropZone)”的概念，用户明确知道存放与提取的边界。
- 或者在平板端打包独立的原生 App 以获取最高的底层权限。

## 🗄️ 档案 7：全景音频路由 (Feature 8) 战略性封存

**业务复盘**：尝试在 C# 层利用 NAudio 劫持 Scrcpy 音频流和 WASAPI Loopback 打造“全景数字调音台”，实现无回声的多设备双响与网络透传。

**封存原因**：Windows 底层 WASAPI 共享模式的进程互斥、Scrcpy stdout 管道的对齐死锁、以及不同设备重采样带来的极高复杂度，导致投入产出比极低。在不使用专业虚拟声卡驱动（如 Voicemeeter）的情况下，纯靠 C# 应用层极难完美填平这些物理天堑。

**处理决定**：果断止损，进行战略性归档。未来的音频方案应回归系统原生逻辑，或引导用户使用专业音频路由软件，不再在业务代码中死磕底层音频管线的分发。
